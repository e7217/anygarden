# Doorae 멀티에이전트 채팅 서버 -- 아키텍처 설계 종합 보고서

> **작성일**: 2026-04-06 (최종 재작성: 2026-04-08) | **작성자**: 수석 리더 | **범위**: 9개 에피소드(경량 서버 + 네이티브 프로토콜 + 채널 기반 서브에이전트)

---

## 1. 프로젝트 개요

### 목적

실제 사용자와 AI 에이전트가 하나의 채팅방에서 자연스럽게 상호작용하는 **멀티에이전트 채팅 서버**의 아키텍처를 결정하기 위해, **9가지 "경량 서버 + 네이티브 프로토콜" 변형**을 설계하고 비교 평가했다. 단일 정답을 찾는 것이 아니라, 동일한 경량 패러다임 하에서 **통신/저장/배포 패턴**이 어떻게 서로 다른 제품 가치를 만드는지 정직하게 드러내는 것이 이 작업의 핵심 목표다.

### 공통 원칙 (모든 9개 에피소드에 적용)

1. **서버는 경량 메시징 허브**: 메시지 라우팅 + 영속화 + 인증/권한만 책임진다. LLM 호출, 도구 실행, 서브에이전트 오케스트레이션 같은 "두뇌 로직"은 서버에 **존재하지 않는다**.
2. **에이전트의 두뇌는 각 엔진**: Claude Code SDK, Codex SDK, OpenHands, Deep Agents 등 **에이전트 엔진이 LLM·도구·메모리를 자체 관리**한다. 서버는 엔진 내부에 간섭하지 않는다.
3. **채팅 연결은 네이티브 프로토콜, MCP는 외부 도구 영역**: 채팅 메시지는 WebSocket / SSE / gRPC / NATS 등 **에피소드별 네이티브 프로토콜**로 전달되고, 런타임 hook이 수신 메시지를 LLM 대화 컨텍스트에 자동 주입한다. **MCP는 별개 영역**이며, 에이전트가 GitHub·Jira·Filesystem 같은 외부 시스템에 접근할 때만 사용한다. 채팅을 MCP 도구로 모델링하면 메시지당 수백 토큰의 도구 정의·호출 오버헤드가 발생하므로 **명시적으로 거부**한다.
4. **서브에이전트 = 채널 기반**: "부모-자식 위임"은 에이전트 엔진의 spawn 기능이 아니라, **두 에이전트가 공유하는 작은 Room(채널)**으로 표현한다. Room 테이블에 `parent_room_id` 컬럼 한 개만 추가하면 되며, 모든 에이전트는 동등한 peer다. 어떤 엔진이든 동일하게 동작한다.
5. **이종 엔진 혼용**: 같은 Room 안에 Claude Code SDK 에이전트, Codex 에이전트, OpenHands 에이전트가 **동시에** 공존할 수 있다. 서버는 엔진의 종류를 알 필요가 없다.

### 범위

- **9개 아키텍처 에피소드**: EP00-05, EP07, EP09, EP10 (EP06·EP08은 폐기 — 사유는 §4)
- **설계 진화**: 초안 → 4회 critic 리뷰(R1·R2·R3·R5) + 5회 개선 → EP10 후속 추가 → 경량 패러다임 전면 통일 재작성 → **MCP 프레이밍 제거 + 채널 기반 서브에이전트로 최종 통일**
- **공통 문서 구조**: 에피소드당 10개 파일(README + 9개 주제별 문서)
- **총 문서량**: 리뷰 5건 + SUMMARY 1건 + 9개 에피소드 × 10파일 = **96개 마크다운 파일**

## 2. 에피소드 일람 (11개)

모든 에피소드가 **같은 경량 원칙**을 공유한다. 차이는 **채팅 연결을 어떤 네이티브 프로토콜로 처리하는가**에 있다. 어떤 에피소드도 채팅을 MCP 도구로 모델링하지 않는다 — MCP는 외부 도구(GitHub, Jira 등) 영역으로 분리되어 있다.

| EP | 채팅 연결 프로토콜 | 핵심 차별점 | 서버 코드 추정 | 이상적 사용 시나리오 |
|:--:|---|------------|:--:|------------|
| **00** | **WebSocket** | 유저와 에이전트 모두 WebSocket으로 같은 프레임 포맷 사용. 가장 직접적인 양방향 실시간 | ~700줄 | 웹 UI 중심의 실시간 채팅 제품. 브라우저 유저와 백엔드 에이전트가 동일 프로토콜로 공존 |
| **01** | **NATS JetStream 직접 구독** | 에이전트가 `doorae.rooms.{id}` 토픽을 직접 구독하여 push 방식으로 메시지 수신. Room = topic | ~800줄 | 수평 확장이 필수인 프로덕션 환경. 저지연 push, 수백 Room 수용. 에이전트가 폴링하지 않음 |
| **02** | **gRPC 양방향 스트리밍** | Protobuf 타입 안전 + HTTP/2 멀티플렉싱. 모든 메시지를 append-only Event Store에 불변 이벤트로 기록 | ~700줄 | 금융·의료·법률 등 **감사 추적이 법적 요구사항**인 환경. 시간 여행 디버깅, 강타입 API |
| **03** | **SSE + REST (Matrix 스타일 이벤트)** | Matrix `/sync` long-polling 패턴을 SSE로 재구성. 이벤트 모델 차용. 선택적 federation 어댑터로 다중 조직 연동 | ~800줄 | 다중 조직/회사 간 에이전트 협업. 본사-자회사, 컨소시엄, 파트너사 Room 공유 |
| **04** | **doorae-sdk + WebSocket (OS 프로세스 격리)** | 각 에이전트 = systemd unit / Docker 컨테이너. SDK가 WebSocket을 흡수, 개발자는 콜백만 작성 | ~570줄 + SDK | 에이전트가 민감한 코드를 실행하거나 격리가 중요한 환경. 보안 샌드박싱, 리소스 쿼터 |
| **05** | **SSE + REST POST** | HTTP 표준만. `POST /rooms/{id}/messages` 송신, `GET /rooms/{id}/stream` SSE 수신. curl 한 줄로 검증 | ~680줄 | 방화벽 제약(WebSocket 차단), 프록시·CDN 친화. 초기 개발/디버깅 생산성 최우선 |
| **07** | **graphql-ws (Subscription)** | GraphQL Subscriptions over WebSocket. 스키마가 데이터 모델 + API 명세 + 문서 역할 | ~1000줄 | 대규모 웹 프론트엔드 팀. 복잡한 UI 상태 관리. React/Apollo 생태계 이미 도입된 경우 |
| **09** | **4종 프로토콜 하이브리드** | WebSocket + SSE/REST + gRPC + NATS 어댑터가 한 서버에서 동시 동작. 통합 메시지 버스로 프로토콜 간 브릿지 | ~1080줄 | 점진적 도입(REST → SSE → WS → gRPC → NATS) 엔터프라이즈. 서버리스/모바일/IoT/레거시 복합 환경 |
| **10** | **WebSocket + doorae-sdk (참조 구현)** | 가장 단순한 경량 참조. 다른 EP의 "기준선". SDK가 런타임 통합을 자동 처리 | ~550줄 | MVP/프로토타입/소규모 팀. 1-2주 내 작동하는 서버. 이후 필요에 따라 다른 EP 패턴 차용 |

> **EP06과 EP08은 폐기됨** — 사유는 §4 설계 진화 과정 참조.

## 4. 설계 진화 과정

### 타임라인

1. **초안 (2026-04-05)**: "서버 중심 아키텍처 10가지"를 탐색. 각 에피소드는 WebSocket/NATS/gRPC/Matrix/Actor/REST/P2P/GraphQL/MCP/Hybrid 등 서로 다른 **서버 구현 패러다임**을 제안.
2. **R1 ~ R3 (2026-04-05 ~ 04-06)**: 3회 critic 리뷰 + 3회 개선. 에피소드별 피드백 31개 항목, 오케스트레이션/에러 복구/보안/가시성 섹션 추가. 9개 공통 섹션 구조 확립.
3. **R4 (2026-04-06, 개선 전용)**: critic 리뷰 없이 R3의 지시대로 구조 통일. 목차 일관성, 교차 참조, 제목 계층 정리. 세부는 [review-round-04.md](review-round-04.md) 참조.
4. **R5 최종 검수 (2026-04-06)**: 최종 리뷰 및 순위 산정. EP09(하이브리드) 1위, EP02(gRPC+Event Sourcing) 2위 등 "서버 중심 관점"의 평가 완료.
5. **EP10 추가 (2026-04-06)**: 관찰 — EP00-09의 서버가 에이전트 엔진의 자체 내장 기능과 상당 부분 중복된다. EP10은 "서버는 메시징 허브, 에이전트의 두뇌는 각 엔진이 책임진다"는 새 패러다임을 제안. 서버 구현량을 ~550줄로 극단적 축소.
6. **경량 패러다임 전면 통일 (2026-04-07, 1차 통일)**: EP00-09를 경량 패러다임으로 재작성. 이때는 "채팅 서버 = MCP 서버 중 하나"라는 프레이밍을 채택하여 모든 에이전트 통신을 MCP 도구 호출로 모델링했다.
7. **MCP 프레이밍 제거 (2026-04-08, 2차 통일)**: 1차 통일의 치명적 결함 발견. 채팅을 MCP 도구로 모델링하면 메시지당 ~200-700 토큰의 도구 정의·호출·결과 오버헤드가 발생하여 LLM 컨텍스트를 잠식한다. **MCP는 외부 도구(GitHub, Jira, Filesystem) 영역으로 분리**하고, 채팅 연결은 각 에피소드의 **네이티브 프로토콜**(WebSocket / SSE / gRPC / NATS / graphql-ws / SDK)로 처리하도록 전면 재작성. 토큰 오버헤드가 ~10-30 토큰으로 **10-30배 절감**됨.
8. **EP06·EP08 폐기 (2026-04-08)**:
   - **EP06 (P2P WebRTC)**: TURN 릴레이가 결국 중앙 서버 의존성을 되살리는 모순. 사용자가 선택한 5가지 연결 방법(WebSocket/NATS/gRPC/SSE+REST/SDK)에 P2P가 포함되지 않음. 기획 폐기.
   - **EP08 (MCP-as-Protocol)**: 채팅을 MCP 도구로 모델링하는 본질이 #7의 결함과 직결. EP08의 핵심 가치 자체가 무효화되어 폐기.
9. **채널 기반 서브에이전트 도입 (2026-04-08, 3차 통일)**: 서브에이전트를 엔진 spawn으로 모델링하던 방식의 복잡도(런타임별 통합, 엔진 종속) 발견. **서브에이전트 = 두 에이전트가 공유하는 작은 Room(채널)** 으로 재정의. Room 테이블에 `parent_room_id` 컬럼 한 개만 추가하면 되며, 모든 런타임에서 동일하게 동작. 9개 에피소드의 `03-agent-hierarchy.md` 일괄 수정.

### 핵심 관찰

세 번의 통일 라운드를 거치며 얻은 가장 큰 깨달음:

1. **레이어를 혼동하지 말 것** — 채팅(메시지 교환)과 도구 호출(외부 시스템 조작)은 서로 다른 레이어다. 둘을 같은 프로토콜(MCP)로 묶으면 추상화 이득보다 컨텍스트 비용이 훨씬 크다.
2. **서브에이전트 = 채널** — 엔진 spawn 방식은 런타임 종속을 만든다. 채널 기반 방식은 기존 Room 인프라를 재사용하면서 런타임 독립, 이종 엔진 혼합, 재사용성, 감사 추적 모두를 자연스럽게 얻는다.
3. **선택의 축이 "통신 프로토콜"로 명확해짐** — 서버의 두뇌 로직이 사라지고 MCP 프레이밍이 제거되면서, 각 에피소드의 본질적 차이는 **"채팅 메시지를 어떤 프로토콜로 전달하는가"** 하나로 수렴했다.

## 7. 에이전트 계층 구조 (채널 기반 — 공통)

**모든 에피소드에서 서브에이전트는 두 에이전트가 공유하는 작은 Room(채널)으로 표현된다**. 엔진이 spawn하는 자식 프로세스가 아니라, 채팅 서버의 Room 하나가 곧 위임 관계를 나타낸다.

### 데이터 모델

```sql
-- Room 테이블에 컬럼 두 개만 추가
ALTER TABLE rooms ADD COLUMN parent_room_id UUID REFERENCES rooms(id);
ALTER TABLE rooms ADD COLUMN room_type VARCHAR(20) DEFAULT 'main';
-- room_type: 'main' (일반 Room) | 'sub' (서브에이전트 전용 채널) | 'dm' (1:1)
```

Agent 테이블에는 `parent_agent_id`가 **없다**. 모든 에이전트는 동등한 peer이며, 관계는 Room 단위로만 표현된다.

### 동작 예시

```
메인 Room: #sprint-42  (room_type=main, parent_room_id=NULL)
├── 참여자: Alice(User), PM, TechLead, Designer, Backend, Frontend
│
├── 서브 채널: #sprint-42/pm-designer-private  (parent=#sprint-42)
│   ├── 참여자: PM, Designer
│   └── 목적: 디자인 작업 위임
│
└── 서브 채널: #sprint-42/techlead-team  (parent=#sprint-42)
    ├── 참여자: TechLead, Backend, Frontend
    └── 목적: 기술 작업 조율
```

부모 Room에는 모든 멤버가 보이지만, 자식 Room은 격리되어 있다. PM과 Designer가 대화하는 내용은 메인 Room의 다른 참여자에게 보이지 않는다.

### 이 방식의 장점

1. **`spawn_sub_agent()` API 불필요** — 기존 `create_room()` + `add_participant()` 재사용
2. **런타임 독립** — Claude Code SDK, Codex, OpenHands, Deep Agents 모두 동일하게 동작. 엔진별 spawn API에 의존하지 않음
3. **이종 엔진 혼합** — Claude Code 부모가 Codex 자식에게 위임 가능. 같은 채널에 참여만 하면 됨
4. **서브에이전트 재사용** — Backend 에이전트 하나가 여러 TechLead의 채널에 동시 참여 가능
5. **장애 격리** — 부모 프로세스가 죽어도 자식은 독립 프로세스이므로 영향 없음
6. **감사 추적** — 모든 위임 대화가 채널에 기록되어 자연스럽게 감사 가능
7. **권한 상속** — 자식 Room은 부모 Room의 프로젝트 스코프와 권한을 자동 상속

### 에피소드별 적용

각 에피소드는 자신의 네이티브 프로토콜로 서브 채널 생성·구독을 처리한다.

| EP | 채널 생성 방법 | 자식 채널 구독 |
|----|--------------|--------------|
| 00 | WebSocket `create_room` 프레임 | 동일 WS 연결로 새 Room 멀티플렉싱 |
| 01 | NATS request-reply (`doorae.rooms.create`) | 새 토픽 `doorae.rooms.{sub_id}` 구독 |
| 02 | gRPC `CreateRoom` RPC | gRPC stream에서 새 Room 메시지 수신 |
| 03 | `POST /api/rooms` (parent_room_id 포함) | 새 Room SSE 스트림 시작 |
| 04 | `client.create_room()` SDK 호출 | SDK가 자동 처리 |
| 05 | `POST /rooms` (parent_room_id 포함) | 새 Room SSE 스트림 시작 |
| 07 | `createRoom` GraphQL Mutation | 새 `messageAdded(roomId)` Subscription |
| 09 | 4종 프로토콜 중 어느 것이든 가능 | 부모와 자식이 다른 프로토콜 가능 |
| 10 | `client.create_room()` SDK 호출 | SDK가 자동 처리 |

핵심: 이 모든 방식이 **같은 Room 시스템 위에 얹힌 얇은 어댑터**일 뿐이다. 서버 내부에는 단 하나의 `RoomService.create_room()` 함수만 있고, 9가지 프로토콜 어댑터가 그것을 호출한다.

## 8. 실질 1순위 추천: EP10 (경량 참조 구현)

현 시점에서 **실질적 1순위는 EP10**이다.

### 근거

1. **가장 단순하다**: ~550줄의 서버 코드. 1-2주 내 MVP 가능.
2. **경량 원칙의 가장 순수한 구현**: EP10은 경량 패러다임의 "기준선"이며, 다른 모든 에피소드는 EP10에 특정 패턴을 덧붙인 변형으로 이해할 수 있다.
3. **마이그레이션이 쉽다**: 수평 확장이 필요해지면 EP01의 NATS 패턴을 차용, 감사 추적이 필요해지면 EP02의 Event Store 패턴을 차용, 멀티 프로토콜이 필요해지면 EP09의 어댑터 패턴을 차용 — 이런 식으로 **필요에 따라 점진적으로 진화**할 수 있다.
4. **이종 엔진 혼용을 즉시 지원**: 서버가 엔진 종류를 모르기 때문에 Claude Code SDK + Codex + OpenHands가 같은 Room에서 자연스럽게 공존.
5. **doorae-sdk가 복잡성을 흡수**: 에이전트 개발자는 5-10줄 코드로 채팅에 참여 가능. WebSocket·재연결·Room 구독·런타임 hook 모두 SDK가 처리.

### 한계 및 차용 전략

| 언제 한계에 부딪히나 | 어느 에피소드 패턴을 차용할까 |
|---------------------|----------------------------|
| Room 수 수백 개 이상, 서버 인스턴스 수평 확장 필요 | **EP01**의 NATS JetStream 직접 구독 패턴 |
| 감사 추적 법적 요구 / 시간 여행 디버깅 | **EP02**의 gRPC + Append-only Event Store 패턴 |
| 외부 조직 연동 | **EP03**의 SSE + Matrix federation 어댑터 |
| 에이전트 샌드박싱 필요 | **EP04**의 systemd/Docker 프로세스 격리 |
| 방화벽/CDN 친화 HTTP 표준 | **EP05**의 SSE + REST POST 패턴 |
| 풍부한 웹 UI 쿼리 | **EP07**의 graphql-ws 패턴 |
| 4종 프로토콜 동시 지원 | **EP09**의 멀티 어댑터 + 통합 메시지 버스 |

## 9. 남은 과제

경량 패러다임으로 통일했지만, 다음 과제들은 여전히 미해결이다.

- **LLM Provider 추상화**: Claude/OpenAI/Gemini/로컬 모델 등 엔진별 LLM 호환성. 어느 층위에서 이 추상화가 이루어져야 하는가? (엔진 레벨? 별도 게이트웨이?)
- **에이전트 성향/프리셋 배포**: 엔진별로 성향 정의 포맷이 다르다(Claude Code의 `CLAUDE.md`, Deep Agents의 시스템 프롬프트 등). 공통 포맷과 변환 계층이 필요한가?
- **SKILL 프리셋 레지스트리**: SKILL.md 파일의 저장·검색·버전 관리. 에이전트가 동적으로 SKILL을 발견하는 표준 프로토콜.
- **프로토타입 검증**: MCP `resources/subscribe`를 통한 실시간 스트리밍 지연 측정. 이종 엔진 혼용 시 대화 품질(턴 충돌, 응답 일관성) 실증.
- **엔진 통합 테스트**: Claude Code SDK + Codex + OpenHands가 같은 Room에서 동시 동작하는 end-to-end 테스트 시나리오.
- **EP별 프로토타입**: 본 보고서는 설계 문서다. 각 EP의 실제 구현 프로토타입(최소 EP10, EP01, EP02)이 필요하다.

## 11. 리뷰 문서 해석 시 주의

- **R1 ~ R5 리뷰 문서는 "서버 중심 설계 시절"의 기록이다**. 경량 패러다임으로 통일되기 **이전**의 평가이므로, 현재의 에피소드 본문과는 세부 내용이 맞지 않을 수 있다.
- 특히 **R5 최종 검수의 순위표(EP09 1위, EP02 2위, EP04 3위 등)는 "서버가 두뇌 로직을 가진다"는 전제 하에서 평가한 것**이다. 경량 패러다임에서는 에이전트 엔진이 두뇌를 책임지므로, EP04의 Supervision Tree 우수성이나 EP02의 Event Store 구조적 이점 같은 평가는 **의미가 달라지거나 약화된다**.
- **EP06과 EP08은 R5 시점에 존재했지만 이후 폐기되었다**. 리뷰 문서에 등장하는 EP06/EP08 평가는 더 이상 유효하지 않다.
- 현재 시점에서 실용적 의사결정이 필요하면 **§6 사용 시나리오별 추천**과 **§8 실질 1순위 추천**을 우선 참조할 것.
- 리뷰 문서들은 **설계 진화의 역사적 기록**으로서 가치를 유지한다. 아키텍처 의사결정의 근거, 포기된 대안, critic 피드백의 유형 등은 여전히 참고할 가치가 있다.
