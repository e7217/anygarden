# 09 · 기존 에피소드 및 Plan A/B/C와의 관계

> 이 구현은 **진공 상태에서 나온 것이 아니다**. 9개 탐색 에피소드 → 3개 최종 안(A/B/C) → 3회 피드백 사이클 위에서 "가장 단순한 구체 구현"을 고른 결과다. 이 문서는 그 계보를 명확히 하고, 언제 이 구현이 아닌 다른 것을 선택해야 하는지를 말한다.

## 9.1 설계 계보

```
9개 탐색 에피소드 (EP00~EP10, EP06/08 폐기)
         ↓
  analyst가 10개 차원 D1~D10으로 분해
         ↓
  3개 최종 안 — 차원 조합이 서로 다른 3개 최적해
         ↓
   Plan A         Plan B         Plan C
  (단순/MVP)   (감사/규제)    (유연/다환경)
         ↓
   R1·R2·R3 교차 리뷰 (치명적 이슈 해결)
         ↓
  [이 구현] ← Plan A + 사용자 요구사항(uvx/바이너리/Docker 없음)
```

이 구현은 **Plan A의 구체적 Python 스택 결정**이다. Plan A 본문(`episodes/final/proposal-a-simple.md`)의 모든 아키텍처 결정은 이 문서의 전제이며, 이 문서는 거기에 "**어떤 패키지, 어떤 파일, 어떤 명령어**" 층위를 덧붙인다.

## 9.2 Plan A/B/C와의 관계

### 9.2.1 Plan A: 이 구현의 직접 부모

| 항목 | Plan A 결정 | 이 구현에서의 구체화 |
|---|---|---|
| 채팅 프로토콜 | WebSocket | FastAPI `WebSocket` + `websockets` Python lib |
| 저장소 | SQLite (→ PG) | SQLAlchemy 2.0 async + aiosqlite + Alembic |
| 인증 | JWT(유저) + API Token(에이전트) | python-jose HS256 + argon2 해시 |
| 서브에이전트 | `parent_room_id + is_dm` | SQLAlchemy 모델로 직접 매핑 |
| LOC 예산 | 필수 ~710 / 옵션 ~790 / 상한 ~960 | 동일 (03/08 문서에서 재확인) |
| MCP | 외부 도구 영역만 | 서버 코드에 MCP 의존성 없음 |

**차이점**: 이 구현은 Plan A의 "추상적 결정"을 "**구체적 pyproject.toml과 CLI와 디렉토리 레이아웃**"으로 내린다. 특히:

- Plan A는 "uvx 배포"라고만 적었지만, 이 구현은 `[project.scripts]`와 첫 실행 UX와 설정 파일 경로를 확정한다
- Plan A는 "Python SDK"라고만 적었지만, 이 구현은 `doorae-sdk` 패키지 구조와 엔진별 optional-dependencies 분리를 확정한다
- Plan A는 "Prometheus 지표"라고만 적었지만, 이 구현은 5개 지표의 메트릭 이름과 라벨과 버킷을 확정한다

### 9.2.2 Plan B: 감사/규제 시 갈 곳

| 항목 | 이 구현 | Plan B |
|---|---|---|
| 채팅 프로토콜 | WebSocket | SSE + REST POST |
| 저장소 | SQLite 일반 테이블 | PostgreSQL append-only Event Store |
| DB 권한 분리 | 없음 | INSERT-only / SELECT-only 이원화 |
| 이벤트 타입 | 단일 Message 테이블 | 13개 이벤트 타입 + projection view |
| 배포 | 단일 프로세스 uvx | systemd/OS 프로세스 격리 필수 |
| 마이그레이션 | 없음 (스키마 변경만) | upcasting chain |
| 코드량 | ~800-950 | ~1,130-1,720 |
| MVP 기간 | 2-3주 | 6-8주 |

**이 구현이 Plan B로 갈 시점**: §7.5.2 (감사/규제 요구 신호)의 트리거가 오면. 마이그레이션 비용은 Plan A 문서 §13.3에 상세 (~1,200-1,500줄 추가, 6-8주).

**핵심**: 감사가 **나중에** 요구되면 Plan B로의 마이그레이션이 매우 비싸다. 감사가 **처음부터** 확정이라면 바로 Plan B로 시작하는 것이 옳다.

### 9.2.3 Plan C: 수평 확장/멀티 프로토콜 시 갈 곳

| 항목 | 이 구현 | Plan C |
|---|---|---|
| 채팅 프로토콜 | WebSocket 단일 | WS + SSE + gRPC + NATS 4종 |
| 메시지 버스 | 서버 내부 (직접) | `MessageBus` Protocol (InProcess / NATS) |
| 수평 확장 | 불가 (단일 프로세스) | NATS 클러스터 기반 |
| 다중 인스턴스 | 불가 | 가능 (Phase 4+) |
| 마이그레이션 경로 | — | Phase 1~5 단계적 |
| 코드량 | ~800-950 | ~1,120-1,540 (Phase 별) |

**이 구현이 Plan C로 갈 시점**: §7.5.1 (수평 확장 신호) 트리거.

**마이그레이션 난이도**: Plan A → Plan C 이동은 Plan A → Plan B 이동보다 훨씬 쉽다 (~550줄, 2-3주). 왜냐하면:

1. Plan C의 Phase 1이 사실상 Plan A와 유사 (MessageBus Protocol 추상화만 추가)
2. 데이터 모델이 호환 (Plan C도 같은 7개 엔티티 + `parent_room_id + is_dm`)
3. WebSocket 어댑터는 재사용 가능

### 9.2.4 "이 구현 ∥ Plan B 공존" 시나리오

Plan B와 이 구현을 동시에 운영하는 것이 유일한 해법인 경우가 있다:

- 일반 채팅(대다수 Room) → 이 구현 사용
- 감사 대상 채팅(컴플라이언스 Room) → Plan B 사용

두 시스템이 독립적으로 돌고, 사용자는 Room의 메타데이터에서 "감사됨" 플래그를 본다. 마이그레이션이 아니라 **병행 운영**.

이 경우 Plan B의 Matrix federation 어댑터를 활용해 두 시스템 간 메시지를 조건부 복제할 수도 있다. 하지만 이는 고급 패턴이며 본 구현의 초기 범위가 아니다.

## 9.5 언제 이 구현 대신 다른 것을 쓰는가

### 9.5.1 결정 트리

```
Q1. 감사/규제/컴플라이언스가 법적 요구인가?
    │
    ├─ YES → Plan B (SSE+REST + Event Store + OS 격리)
    │        처음부터 Plan B로 시작. 마이그레이션 비용이 크기 때문에.
    │
    └─ NO ↓
Q2. 4종 프로토콜(WS/SSE/gRPC/NATS) 중 2개 이상을 동시 지원해야 하는가?
    │
    ├─ YES → Plan C (하이브리드 멀티 프로토콜)
    │        단, Phase 4(NATS 백본)까지 갈 계획이 없다면 이 구현으로 시작 후 필요 시 마이그레이션
    │
    └─ NO ↓
Q3. 단일 VPS에서 단일 프로세스로 2-3주 내 MVP가 목표인가?
    │
    ├─ YES → ★ 이 구현 ★
    │        Plan A 기반 Python 스택, uvx 배포
    │
    └─ NO ↓
Q4. Python 런타임조차 깔지 못하는 환경(임베디드, 제한된 OS)에 배포해야 하는가?
    │
    ├─ YES → Go 서버 rewrite (4-6주 투자, 단일 바이너리 10-20MB)
    │        단, Python SDK는 그대로 유지 (에이전트 엔진 생태계는 Python 편향)
    │
    └─ NO → 이 구현의 PyInstaller 옵션(§8.4.3)
```

### 9.5.2 신호 매트릭스

| 신호 | 원인 진단 | 대응 |
|---|---|---|
| p99 메시지 라우팅 > 100ms 지속 | 단일 프로세스 한계 | Plan C 수평 확장 차용 |
| 동시 WS 연결 > 500 | 같음 | 같음 |
| "감사 로그 요구" 문서 수신 | 규제 대응 | Plan B 즉시 평가 |
| "외부 조직과 Room 공유" 요청 | 연동 요구 | Plan B federation 또는 Plan C 조합 |
| "모바일 앱 + 서버리스 + 브라우저 동시 지원" | 이종 클라이언트 | Plan C 필요 |
| "Python 없는 VPS에 배포" | 배포 환경 | PyInstaller 또는 Go rewrite |
| "Python 없는 라우터/임베디드 기기" | 극한 환경 | Go rewrite 확정 |

## 9.6 의도적으로 하지 않은 것

### 9.6.1 서버에서 제외한 것

- **LLM 호출**: 각 에이전트 엔진 책임
- **MCP 도구 실행**: 각 에이전트 엔진 책임
- **오케스트레이션 로직**: Host 에이전트(자연어 프롬프트) 위임
- **에이전트 라이프사이클 관리**: systemd/supervisor 위임
- **다중 인스턴스 동기화**: 단일 프로세스 전제
- **감사 로그 별도 저장**: Plan B 영역
- **이벤트 소싱/CQRS**: Plan B 영역
- **GraphQL/gRPC/NATS**: Plan C 영역
- **Matrix federation**: Plan B 옵션 영역
- **WebRTC/P2P**: 폐기 (EP06)
- **채팅을 MCP 도구로 모델링**: 폐기 (EP08)

각각을 구현하지 않은 이유는 이 구현의 **범위 정의**다. 이것들이 필요해지면 해당 Plan으로 이동하거나 이 구현을 확장해야 한다.

### 9.6.2 SDK에서 제외한 것

- **자체 LLM 호출 로직**: 각 엔진 어댑터가 엔진의 공식 SDK를 사용. 직접 API를 호출하지 않음.
- **MCP 클라이언트 재구현**: 각 엔진이 이미 MCP 클라이언트를 내장하므로 SDK는 개입 안 함.
- **자체 메시지 큐**: 서버가 `Last-Seq`로 복구를 제공하므로 SDK는 단순 WebSocket 래퍼.
- **자체 인증 서버**: 토큰은 서버가 발급, SDK는 사용만.
- **자체 프로필 시스템**: Doorae의 기존 `AgentProfile`을 재사용하거나 간단한 YAML 로더만 제공.

**원칙**: SDK는 "채팅 서버에 연결하는 가장 얇은 래퍼"이지, 에이전트 프레임워크가 아니다.

## 9.7 철학적 요약

이 구현의 **5가지 핵심 신념**:

1. **서버는 얇게** — 메시지 라우팅 + 영속화 + 인증. 그 이상은 다른 계층의 몫.
2. **채팅은 네이티브 프로토콜로** — WebSocket이 충분하다. MCP는 외부 도구 영역에 머물러야 한다.
3. **에이전트의 두뇌는 엔진에** — LLM, 도구, 오케스트레이션 로직을 서버에 넣지 않는다. Claude Code SDK, Codex, OpenHands, Deep Agents가 이미 훌륭하게 해낸다.
4. **서브에이전트는 채널** — 엔진 spawn이 아니라 `parent_room_id`로 묶인 작은 Room이다. 이래서 모든 엔진이 동일하게 동작한다.
5. **배포는 `uvx` 한 줄로** — Docker도, Kubernetes도, nginx reverse proxy도 필요 없다. 필요하면 그것은 이 구현이 아닌 Plan B나 Plan C다.
6. **Machine은 1급 리소스, 하지만 경량** — `doorae-machine` Daemon(§10)이 각 머신에서 subprocess spawn을 관리한다. 채팅 평면과 제어 평면은 분리된다. Docker는 여전히 없다.

이 6가지를 지키면 **4-5주 안에 MVP가 동작한다**. 이것이 본 구현의 약속이다.

## 9.8 Plan A 대비 한 계층 추가 (Machine 스케줄링)

이 구현은 Plan A의 모든 원칙을 계승하되, 사용자의 명시적 요구("호스트당 복수 에이전트 + 프로젝트와 머신 독립 + SDK로 원격 접근")를 충족하기 위해 **Machine 스케줄링 계층**을 추가했다. Plan A 원본은 Machine을 수동 메타데이터로만 취급했지만, 이 구현은 Machine을 **1급 스케줄링 리소스**로 격상한다.

| 측면 | Plan A 초안 | 이 구현 (§10 포함) |
|---|---|---|
| Machine 역할 | DB 테이블의 수동 메타데이터 | 등록/감지/선택/명령 대상의 1급 리소스 |
| 에이전트 기동 | 사용자가 `uvx doorae-agent` 수동 실행 | `POST /api/v1/agents`로 선언적 생성 → Daemon이 자동 spawn |
| 다중 머신 운영 | 1-3 머신 로컬 테스트 한정 | N대 머신, bin-pack 자동 배치 |
| 크래시 복구 | systemd가 수동 재시작 | Daemon이 `restart_policy`로 자동 재시작 |
| 엔진 감지 | 수동 설정 | Daemon이 자동 감지 (claude-code/codex/openhands/deep-agents/openai/anthropic) |
| 서버 LOC | ~710 | ~1,330-1,750 (Machine 스케줄러 +~370-790, 01 §1.10.4 / 10 §10.13) |
| 총 패키지 | 2개 (server, sdk) | 3개 (server, sdk, **machine**) |
| MVP 기간 | 2-3주 | 4-5주 |
| Docker 필요 | 없음 | 여전히 없음 |

이 계층 추가는 Plan A의 "900-1,100 LOC" 경계를 명확히 넘고, Plan C 원안의 상한(~1,540)도 소폭(약 200줄) 초과한다 (10 §10.13 정본: `~1,750`). 즉 이 구현은 Plan A와 Plan C의 **중간보다 약간 Plan C 상한을 넘어선 지점**에 위치한다. 그럼에도 경량 철학은 유지된다 — NATS 백본이나 멀티 프로토콜 하이브리드는 여전히 없고, 단일 프로세스 + 네이티브 WebSocket + Docker 없음 원칙은 그대로다.

**Standalone 모드는 폴백으로 유지**된다. Machine Daemon 없이도 `uvx doorae-agent`로 1-2 에이전트를 직접 기동할 수 있다. 로컬 테스트와 단발 데모에 사용.

자세한 설계는 [10-machine-scheduler.md](10-machine-scheduler.md) 참조.

## 9.9 최종 한 줄

> **"9개 에피소드 + 3개 안 + 3회 피드백 + Machine 스케줄링 계층 = Plan A를 Python으로 구체화한 가장 단순한 최종 구현, 다중 머신 운영까지 지원."**

이 구현을 선택하는 것은 "지금 빠르게 동작시키고, 필요해지면 옮긴다"는 전략이다. 그 '필요'가 언제 오는지는 §9.5 결정 트리가 판별한다. 그때까지는 이 구현이 가장 옳은 답이다.
