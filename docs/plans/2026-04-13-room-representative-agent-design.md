# 룸 대표 에이전트

## 개요

각 룸에 대표 에이전트를 지정하여, 사용자가 `#룸` 멘션으로 해당 룸에 질문하면 대표 에이전트가 룸 내 다른 에이전트들의 의견을 실시간 수집·취합해 종합 답변을 전달한다.

## 핵심 결정

| 항목 | 결정 | 근거 |
|------|------|------|
| 대표 에이전트 정체 | 기존 에이전트 중 admin이 지정 | 자동 시스템 에이전트는 리소스 부담 + 설정 문제 |
| 의견 수집 방식 | 실제로 다른 에이전트에게 질문 | 히스토리 요약은 새 의견을 못 얻음 |
| 대화 장소 | 타겟 룸에서 직접 | 서브룸 격리는 불필요한 복잡도 |
| 수집 완료 기준 | 전원 응답 + 5분 안전 타임아웃 | 부분 취합 허용 |
| 요청 룸 미참여 시 | 영구 참여 | 임시 참여는 퇴장 관리 복잡, 서버 대리전송은 비정상 경로 |
| 구현 접근법 | 하이브리드 (서버 라우팅 + SDK 오케스트레이션) | 서버가 전달 보장, SDK가 검증된 delegate 패턴 재활용 |

## DB 변경

`rooms` 테이블에 필드 추가:

```
rooms.representative_agent_id  INTEGER  NULL  FK → agents.id  ON DELETE SET NULL
```

- NULL이면 대표 없음 (`#룸` 멘션 시 무반응)
- 대표로 지정된 에이전트는 반드시 해당 룸의 참여자여야 함 (API에서 검증)

## API 변경

### 룸 대표 지정

```
PUT /api/v1/rooms/{room_id}/representative
Body: { "agent_id": "abc123" }    ← NULL 보내면 해제
Response: 200 { room_id, representative_agent_id }
Auth: admin
```

검증: 해당 에이전트가 이 룸의 참여자인지 확인. 아니면 400 에러.

### 룸 조회 응답 확장

기존 `GET /api/v1/rooms/{room_id}` 응답에 `representative_agent_id` 포함.

## 서버 흐름 (WS handler)

사용자가 `#백엔드룸 API 설계 의견?`을 전송하면:

1. **멘션 감지** — 기존 `parse_mentions()`가 `<#room:id>` 파싱 (이미 구현됨)
2. **대표 에이전트 조회** — 해당 룸의 `representative_agent_id` 확인. NULL이면 무시
3. **자동 참여** — 대표가 현재 룸의 참여자가 아니면 Participant로 추가 (영구)
4. **metadata 부착** — `extra_metadata`에 `room_query` 정보 추가:

```json
{
  "mentions": [{"type": "room", "id": "백엔드룸id"}],
  "room_query": {
    "target_room_id": "백엔드룸id",
    "source_room_id": "디자인룸id"
  }
}
```

5. **일반 브로드캐스트** — 메시지 내용은 사용자가 쓴 그대로 저장·전달

## SDK 흐름 (대표 에이전트)

1. **should_respond 확장** — `room_query`가 있고, 자신이 `target_room`의 대표이면 응답
2. **타겟 룸에 질문 전달** — 백엔드룸에 `[ROOM_QUERY] 디자인룸에서 질문: API 설계 의견?` 전송
3. **응답 수집** — 백엔드룸의 다른 에이전트들이 `[ROOM_QUERY]` 접두사를 보고 답변 (기존 `[DELEGATED]`와 같은 패턴). 전원 응답 대기 + 5분 안전 타임아웃
4. **취합** — 모든 응답이 모이면 대표 에이전트가 LLM으로 종합
5. **결과 전달** — 디자인룸(source_room)에 취합 결과 전송

기존 delegate 패턴(`execute_delegate` + 콜백 + 타임아웃)을 재활용하되, 단일 에이전트 응답이 아니라 복수 에이전트 응답 수집으로 확장.

## 전체 시퀀스

```
사용자 (디자인룸)                서버                    대표 에이전트 B (SDK)           백엔드룸 에이전트들
     |                           |                           |                           |
     |-- #백엔드룸 의견? -------->|                           |                           |
     |                           |-- parse_mentions()        |                           |
     |                           |-- 대표 조회 (B)           |                           |
     |                           |-- B를 디자인룸 참여 (영구) |                           |
     |                           |-- metadata에 room_query   |                           |
     |                           |-- 메시지 브로드캐스트 ---->|                           |
     |                           |                           |-- should_respond (room_query 감지)
     |                           |                           |-- 백엔드룸에 [ROOM_QUERY] 질문 전달 -->|
     |                           |                           |                           |-- D 응답
     |                           |                           |                           |-- E 응답
     |                           |                           |<-- 전원 응답 수집 ---------|
     |                           |                           |-- LLM 취합               |
     |<------------------------- |<-- 디자인룸에 종합 답변 ---|                           |
```

## 에러 처리

| 상황 | 처리 |
|------|------|
| `#룸`의 대표 에이전트가 NULL | 무시 — 일반 `#룸` 링크로만 동작 (현재와 동일) |
| 대표 에이전트가 offline (stopped/crashed) | 요청 룸에 "대표 에이전트가 오프라인입니다" 시스템 메시지 |
| 5분 내 일부 에이전트만 응답 | 응답한 에이전트 것만 취합, 미응답 에이전트 명시 |
| 타겟 룸에 대표 외 에이전트가 없음 | 대표가 혼자 답변 (취합 없이 직접 응답) |
| 대표 에이전트가 타겟 룸에서 제거된 상태 | `representative_agent_id` 무효 — NULL과 동일 처리 |
