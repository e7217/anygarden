## Situation

anygarden은 4개 엔진 어댑터(codex-cli, gemini-cli, claude-code, openhands)를 지원하지만, 엔진 CLI의 **최신버전 확인·업데이트** 기능이 없었다. 게다가 한 엔진의 사실(설치 채널·패키지명·감지법)이 3개 패키지·4곳(agent 어댑터 등록, machine detector, cluster 카탈로그, 어댑터별 힌트 문자열)에 흩어져 있어 새 엔진 추가·기능 확장이 어려웠다. 기존 업데이트 인프라인 #546(PyPI 버전 감지)은 anygarden 자체 패키지만, #550(머신 자기-업데이트)은 데몬 자신만 대상이라 엔진 CLI는 어느 쪽도 커버하지 못했다.

## Task

- 엔진 **수명주기**(감지 → 현재버전 → 최신버전 → 업데이트)를 단일 추상으로 응집한다.
- 새 엔진 추가 = descriptor 1항목 (+ 새 설치 채널일 때만 Channel 1개).
- 웹 UI에서 엔진 버전을 확인하고 버튼으로 업데이트한다(cluster + frontend).
- 서버가 임의 패키지명을 주입하지 못하도록 #550의 보안 불변식을 계승한다.
- 기존 인프라(#550 WS 오케스트레이션, #546 비교/UI/캐시)를 재사용한다.

## Action

**machine** — 수명주기 추상 신설:
- `engines/channels.py`: `Channel` Protocol + `NpmGlobal`/`PipVenv`. `latest_version`(registry 조회, 실패시 None), `normalize`(raw `--version`→비교가능 버전), `update_argv`(셸 미경유). 정규화 정규식은 두 채널 공유.
- `engines/registry.py`: `EngineLifecycle`(engine/detect/channel/package) + `ENGINE_LIFECYCLES`(4엔진) + `get_lifecycle`(허용목록 게이트, unknown→None).
- `engines/updater.py`: `run_engine_update` — 허용목록 검증 후 `channel.update_argv` 실행. pip 채널은 명시적 인터프리터 필수(sys.executable 폴백 제거로 잘못된 venv 설치 방지).
- `detector.py`: `BINARY_ENGINES`/`PYTHON_MODULE_ENGINES`를 레지스트리 `detect`에서 파생하도록 재배선(상수 이름 유지 → 하위호환).
- `protocol/frames.py`: `EngineCheckFrame`/`EngineUpdateFrame`(서버→머신) + `EngineCheckResultFrame`/`EngineUpdateResultFrame`(머신→서버) + `_SERVER_FRAME_MAP` 등록.
- `daemon.py`: `_handle_engine_check`(감지+조회+정규화→결과), `_handle_engine_update`(updating→success/failed, `asyncio.to_thread`로 subprocess off-loop).

**cluster**:
- `db/models.py`: `MachineEngineStatus`((machine_id, engine) unique). migration `056`.
- `api/v1/machines.py`: `POST /{id}/engines/{engine}/check|update`(owner-or-admin, offline 409, update는 `update_status="updating"` 기록). `GET /{id}/engines` 응답에 `latest_version`/`update_available`/`update_status` 병합.
- `ws/machine_handler.py`: `_handle_engine_check_result`(→`is_update_available` #546 재사용→저장), `_handle_engine_update_result`(updating→success/failed).

**frontend**:
- `hooks/useMachines.ts`: `checkMachineEngine`/`updateMachineEngine`.
- `components/AdminMachines.tsx`: 엔진 행에 현재/최신버전·`EngineStatusBadge`·Check/Update 버튼(#546 StatusBadge 스타일 이식, DESIGN.md 준수).

## Decisions

설계 근거는 `.tmp/plan-553-engine-lifecycle-abstraction.md` §3.2와 설계 스케치 `docs/plans/2026-07-24-engine-lifecycle-abstraction-design.md`에서.

- **추상화 범위 = 수명주기 레지스트리 + 채널(단수)**. 대안: (a) 이번 기능 전용 레지스트리 — detector와 정보 중복·파편화 유지라 기각; (b) 단일 God descriptor로 4곳 전면 통합 — catalog는 cluster에서 의도적 수동관리이고 어댑터 등록은 런타임 관심사라, 셋을 한 모듈로 묶으면 uv workspace 3패키지가 서로를 import하게 되어 경계가 무너지므로 기각. 결정적 근거: 감지·최신조회·업데이트가 전부 machine 프로세스의 관심사라 machine 안에서 자연히 응집.
- **최신조회 위치 = machine**. 대안: cluster(#546 `version_service`와 대칭). `machine/pyproject.toml`에 `httpx`·`packaging`이 이미 있어 조회+정규화가 자족적이고, "패키지명 허용목록의 소스는 machine"이라는 보안 불변식과 일치하므로 machine 채택. 비교(`is_update_available`)만 cluster에서 재사용.
- **상태 저장 = 별도 `MachineEngineStatus` 테이블**. 대안: `machine_engines` 컬럼 확장 / register upsert 로직 변경. 결정적 근거: `_handle_register`가 register마다 `machine_engines`를 delete+recreate하는데, 업데이트 성공 확인이 바로 그 register의 version 변화로 이뤄지므로 같은 행에 상태를 두면 확인 순간 소실되는 자기모순이 생긴다. (테스트 `test_status_survives_register_wipe`로 검증.)
- **채널 단수** (사용자 확정): claude-code의 npm바이너리+pipSDK 이원성은 "업데이트 대상=감지에 쓰는 바이너리 채널" 하나로 축약.
- **보안**: 서버는 엔진 키만 전달, 머신 `ENGINE_LIFECYCLES`가 엔진→패키지명 허용목록의 유일한 소스(#550 `PACKAGE_NAME` 고정 계승).
- **가정(재검토 트리거)**: claude가 실제로 npm `@anthropic-ai/claude-code`로 설치되는지 라이브 미확인(추정, 레지스트리에 주석 표기). npm semver↔PEP440 정규화는 순수 x.y.z만 호환, pre-release 표기는 엣지케이스. openhands(pip) 업데이트는 agent venv 인터프리터 경로 확보가 전제라 현재는 인터프리터 없으면 실패로 표면화.
